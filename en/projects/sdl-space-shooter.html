SDLSpaceShooter

A 2D space shooter hand-written from scratch in C++17 + SDL2

C++17 SDL2 Game Dev CMake Cross-platform

Background

To really understand how a game "runs", reading engine-wrapped APIs isn't enough — the render loop, frame sync, and object management are exactly what engines like Unity or Godot hide behind the scenes.

So I chose the more "bare" path: with C++17 + SDL2, without any game engine, I wrote a complete 2D space shooter line by line — from window creation and the event loop all the way to rendering.

The project started from ZiyuGameDev's SDL2 tutorial on Bilibili, but I didn't stop at "typing along". On top of the tutorial's skeleton I rewrote several parts of the architecture and added a whole set of my own gameplay and engineering improvements.

Tech Stack

  • Language: C++17
  • Graphics / Events: SDL2
  • Extensions: SDL2_image (PNG textures), SDL2_ttf (TrueType UI & leaderboard rendering), SDL2_mixer (BGM + SFX)
  • Build: CMake (≥ 3.20), cross-platform on Windows (MinGW) / Linux / macOS
  • Scale: ~1400 lines of C++ in src/, plus full art, music, and sound assets

My Role

From zero to a shippable build, entirely on my own: architecture design, gameplay coding, parameter tuning, cross-platform builds, and finally the Windows v1.0 package (unzip & play).

I organized the whole game around two core abstractions:

1. Singleton Game + scene state machine

Game is a singleton via a static local variable, holding the window, renderer, main loop, global score, and leaderboard. The game's phases are split into three scenes, all inheriting from the abstract Scene base class:

  • SceneTitle: title screen
  • SceneMain: core gameplay
  • SceneEnd: results + name input + leaderboard write

Game::changeScene() switches the active scene, and input/update/render are uniformly forwarded to currentScene. Each scene only cares about its own logic, keeping the main loop clean.

2. Data-driven object design

Every game object (player, enemies, both sides' bullets, explosions, drops, background) is a plain-data struct in Object.h — fields like position, velocity, HP, cooldown. Gameplay logic lives in SceneMain, managing each object list's lifecycle with std::list. Separating data from logic makes adding objects or tuning numbers lightweight.

Challenges & Solutions

① Framerate-independent motion
Moving "N pixels per frame" makes the game run at different speeds on 144Hz vs 60Hz machines. The fix is deltaTime: every displacement becomes speed * deltaTime * scale, tying motion to real time instead of frame count — consistent feel on any refresh rate.

② Multi-resolution scaling
Displays vary a lot. I use a single scale factor: the window is generated at 90% of screen height with a fixed 3:4 ratio, then all sprite sizes, movement speeds, and UI elements multiply by that factor. One codebase, no broken proportions on any screen.

③ Where should saves go
The leaderboard was first written to the project root — then a packaged release or re-download lost it. I switched to SDL_GetPrefPath, writing to the system user-data directory (AppData/Roaming on Windows). That's the correct approach — saves follow the user, not the program directory. The leaderboard itself uses std::multimap<int, string, greater<int>>, auto-sorted descending on insert, no manual sort.

④ Making combat "feel" right
The tutorial's original felt flat. I added several gameplay tweaks: enemy bullets compute a direction toward the player to track them; the player fires three-shot bursts forming a horizontal bullet pattern; difficulty ramps over time (spawn rate grows by 3 + (elapsed-60000)/5000 after the first minute, capped at 8); plus sprite-sheet explosion animation, drops bouncing off walls three times, and a two-layer parallax starfield. Stacked together, the feel and pacing change noticeably.

Outcome & Takeaways

  • A 2D game with a complete loop from window to release, unzip-and-play on Windows (dist/SDLSpaceShooter_v1.0.zip).
  • A real grasp of the game loop's essence: input → update (with deltaTime) → render, and why framerate independence matters.
  • Engineering skills that transfer to any project: singleton vs state-machine tradeoffs, data/logic separation, cross-platform CMake config, and the easy-to-trip practical question of "which system directory should saves/config live in".
  • Biggest lesson: following a tutorial is only the start; you truly learn once you begin changing it and adding your own ideas. Balance, tracking bullets, difficulty curves, adaptive scaling — all chewed through beyond the tutorial.
